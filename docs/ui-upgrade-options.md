# Windows UI 升级方案（样式现代化）

日期：2026-04-08

## 背景

当前项目基于 `C++ + Win32`，功能完整，但设置页和主弹窗的视觉层次较弱，整体风格偏“系统默认控件”。

目标是：

- 在不破坏现有系统集成能力（剪贴板、托盘、热键、开机启动等）的前提下，提升 UI 观感
- 控制迁移风险，避免一次性重写
- 允许按阶段交付，而不是长时间冻结功能

## 方案对比

| 方案 | 视觉上限 | 迁移成本 | 体积影响 | 与现有 Win32 集成 | 结论 |
| --- | --- | --- | --- | --- | --- |
| WinUI 3 + Windows App SDK（可配合 XAML Islands） | 高 | 中 | 中 | 高 | 推荐主路线 |
| Qt 6（Widgets/Quick） | 高 | 高 | 中-高 | 中 | 备选（跨平台诉求强时） |
| wxWidgets | 中 | 中 | 低-中 | 高 | 备选（保守改造） |
| Dear ImGui | 中（工具风） | 中 | 低 | 中 | 不建议用于最终消费级桌面 UI |

## 推荐决策

推荐采用：

- `WinUI 3` 作为新 UI 技术栈
- 迁移策略采用“渐进式”，先改设置窗口，再改主弹窗
- 在过渡期继续保留现有 Win32 主循环与系统能力

原因：

- 在 Windows 上能最快获得“现代桌面应用”观感（Fluent 风格、动画、排版）
- 不需要一次性重写业务层，可先改 UI 外壳
- 与系统 API 互操作路径成熟，可逐步替换界面模块

## 分阶段落地计划

### 阶段 1：样式止血（1~2 天）

- 统一字体、间距、分组留白、控件高度
- 改造 Win32 主题色和分割线样式，先把“可读性”和“层次感”拉起来
- 目标：在不引入新依赖的情况下，把“low”感明显压下去

### 阶段 2：设置页 WinUI 化（3~7 天）

- 新建 WinUI 设置窗口（优先覆盖 `General`、`Appearance`）
- 通过桥接层读写现有 `AppSettings`
- 保持现有托盘、热键、剪贴板逻辑不变
- 目标：先让用户最常用配置页变现代

### 阶段 3：主弹窗 WinUI 化（1~2 周）

- 将搜索框、列表、预览区迁移到 WinUI 组件
- 保留现有 core 逻辑（搜索、排序、pin、忽略规则）
- 完成键盘导航、回车粘贴、删除等交互对齐

### 阶段 4：收敛与清理

- 删除老旧 Win32 UI 分支代码（仅 UI 层）
- 补齐 UI 自动化/回归测试用例
- 固化设计规范（字体、颜色、间距、控件状态）

## 代码层建议

- 保持 `src/core` 和 `src/platform/win32` 不变，优先改 `src/app` 的 UI 组织层
- 抽一个 `SettingsViewModel`（或等价层）隔离 UI 与配置结构
- 所有新 UI 事件都走“单向更新 + 显式 Apply”，减少状态错乱

## 风险与规避

- 风险：一次性替换主弹窗导致键盘行为回归  
  规避：先迁设置页，主弹窗后移；主弹窗迁移时做行为对照清单

- 风险：WinUI 过渡期消息循环与焦点行为复杂  
  规避：先做最小 PoC（仅设置页），再扩展到主界面

- 风险：发布体积上升  
  规避：保留 `Win32 minimal` 构建通道，分 release 轨道对比体积和启动速度

2. 并行做阶段 2 的最小 PoC（仅 General 页）
3. 用 PoC 结果决定是否全面转入 WinUI 3
